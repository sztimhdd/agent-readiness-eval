# Task 002 候选方案

## 候选 A：AI 平台生产事件调查

### 题目名称
AI Platform Incident Investigation（AI 平台生产事件调查）

### 业务场景
某企业 AI 平台的客户升级事件。`inputs/` 中包含 5 个文件：
- 客服工单记录（客户报修）
- 部署变更日志
- 运行监控日志摘要
- 客户邮件原文
- 团队内部备忘录（含干扰信息——另一个不相关项目的备注）

Agent 需找出与事件真正相关的文件，建立时间线，识别根因。

### Agent 可见输入
5 个文件，分布在 `inputs/` 目录下：

```
inputs/
├── tickets.json           # 4 条客服工单
├── deployment-log.md      # 最近部署记录
├── system-metrics.txt     # 监控指标摘要
├── customer-email.txt     # 客户投诉邮件
└── team-notes.md          # 内部备注（含干扰）
```

### 要求生成的产物
```
task-id.txt
final-answer.md
run-metadata.json
artifacts/investigation-summary.json
```

### 主要测试能力
- 多文件主动查找（扫描目录、识别相关文件）
- 跨文件交叉核对（工单时间 vs 部署时间 vs 监控异常时间）
- 区分相关与干扰信息（`team-notes.md` 中有一段不相关项目的备注）
- 识别信息缺口（某些阶段没有日志覆盖）
- 结构化输出

### 为什么比 Task 001 更难
Task 001 所有信息在单一 JSON 中，Task 002 需要 agent 先发现 5 个文件、判断哪些相关、跨文件比对时间线和事件。

### 为什么仍然跨 Harness 公平
仅依赖 read directory / read file / write file。不调用任何 Harness 专属工具。无代码执行。

### 可能的评分歧义
- 根因可能有不同解读（多因素叠加时，孰为主因）
- 干扰文件的排除标准可能因 reviewer 不同有细微差异
- 时间线构建中，如果 agent 忽略某个文件但核心结论仍正确，是否扣分

### 实现复杂度
中。需要设计 5 个文件的内容和时间线，确保事件链逻辑自洽，干扰文件不喧宾夺主，根因有多层解释空间但有一个最合理的。

## 候选 B：多项目合规审计

### 题目名称
Multi-Project Compliance Audit（多项目合规审计）

### 业务场景
`inputs/` 中包含 4 个项目的合规自评表和一个整体合规政策文件。部分项目自评完整，部分有缺失，一个项目的自评与其他文件存在冲突。Agent 需跨项目核对合规状态，识别不一致和未达标项。

### Agent 可见输入
```
inputs/
├── policy.md                  # 合规政策要求
├── project-alpha-audit.json   # 项目 Alpha 自评
├── project-beta-audit.json    # 项目 Beta 自评
├── project-gamma-audit.json   # 项目 Gamma 自评
└── project-delta-audit.json   # 项目 Delta 自评（含干扰项——部分字段填错版本号）
```

### 要求生成的产物
```
task-id.txt
final-answer.md
run-metadata.json
artifacts/compliance-report.json
```

### 主要测试能力
- 多文件对比（政策 vs 多个项目自评）
- 一致性检查（发现自评与政策的偏差）
- 发现数据质量问题（delta 自评中有明显填错的数据）
- 优先级排序（哪些不合规项最严重）

### 为什么比 Task 001 更难
从 1 个文件 5 条记录 → 5 个文件需要交叉比对。不仅需要提取信息，还需要对比、发现不一致。

### 为什么仍然跨 Harness 公平
纯文件操作。无专属工具依赖。

### 可能的评分歧义
- 合规判断可能有合理弹性（部分政策条文的解释空间）
- 跨项目排序主观成分较大

### 实现复杂度
中低。题库内容相对结构化（JSON + Markdown），不需要复杂时间线设计。

## 候选 C：发布回溯与根因分析

### 题目名称
Release Rollback Root-Cause Analysis（发布回溯与根因分析）

### 业务场景
某 SaaS 平台一次发布后出现性能退化，不得不回滚。`inputs/` 中包含发布记录、代码变更列表（仅标题和文件路径，无实际代码）、监控指标前后对比、QA 测试报告、值班人员备注。Agent 需判断最可能的根因，区分代码问题、配置问题和环境问题。

### Agent 可见输入
```
inputs/
├── release-manifest.json       # 本次发布包含的变更列表
├── performance-before-after.md  # 监控指标对比
├── qa-test-report.md           # QA 回归测试结果
├── oncall-notes.txt            # 值班人员备注（含主观猜测）
└── config-changes.md           # 配置变更记录（与本次发布无关的另一个变更）
```

### 要求生成的产物
```
task-id.txt
final-answer.md
run-metadata.json
artifacts/root-cause-analysis.json
```

### 主要测试能力
- 多文件信息关联
- 技术根因推断（从症状到原因）
- 区分事实与主观判断（oncall-notes 中有猜测内容）
- 排除无关信息（另一个独立的配置变更）

### 为什么比 Task 001 更难
需要技术推理（性能指标解读、变更影响推断），而不仅仅是统计和排序。

### 为什么仍然跨 Harness 公平
无代码执行。提供的是变更标题而非代码 diff，agent 通过阅读理解而非编译执行来推断。

### 可能的评分歧义
- 根因推断空间较大，不同 reviewer 可能对"最可能根因"有分歧
- 需要 reviewer 有一定技术背景理解性能指标
- "变更有 N 个，哪个是根因"可能缺少足够证据做唯一判断

### 实现复杂度
高。需要设计合理的技术场景，确保根因可从提供的信息中推导且唯一，同时避免变成代码能力测试。

## 评分与选择

| 标准 | 权重 | 候选 A | 候选 B | 候选 C |
|------|:----:|:------:|:------:|:------:|
| 与 Task 001 能力梯度清晰 | 25 | **24** — 单文件→多文件，统计→整合，差距正好一级 | 22 — 也是多文件，但偏结构化比对，梯度略小 | 20 — 需要技术推理，跳跃稍大 |
| 跨 Harness 公平性 | 25 | **25** — 纯文件读写，无争议 | 25 — 同上 | 22 — 性能指标解读需要 reviewer 技术判断 |
| 人工评分确定性 | 20 | **18** — 根因多因素时可能分歧，但可接受 | 20 — 合规判断相对确定 | 14 — 根因判断主观空间较大 |
| 业务真实性 | 15 | **15** — AI 平台事件，贴近目标用户 | 13 — 合规审计场景真实但偏管理 | 14 — 技术回溯真实，但门槛略高 |
| 实现和测试成本 | 15 | **14** — 5 个文件时间线需精心设计 | 15 — 结构化数据，实现简单 | 12 — 需设计可验证的性能场景 |
| **总分** | **100** | **96** | **95** | **82** |

### 选择结果：候选 A — AI 平台生产事件调查

**理由：**
1. 与 Task 001 的能力梯度最清晰：单文件统计 → 多文件整合，正好一个难度等级
2. 跨 Harness 公平性无争议：纯文件读写
3. 评分确定性高：时间线、证据引用、文件相关性都是客观可判的
4. 业务场景真实且贴近目标用户（AI 平台运营）
5. 实现成本中等，时间线逻辑可控

**与 /goal 推荐方向一致**（多文件证据整合与决策），没有发现阻止选择它的公平性或评分问题。

候选 B 总分接近但梯度略小（偏结构化比对而非整合推理）。候选 C 因评分主观性被拉开差距，适合放在更高级别（Task 003）再做技术根因分析。
