# Task 001 → Task 002 能力差距分析

## 概述

Task 001（客服工单分类）测的是单文件信息提取、基础统计和业务排序。Task 002 需要在同一框架下提高一个难度等级，保持跨 Harness 公平，不引入代码执行或复杂工具。

## 能力维度对比

| 能力维度 | Task 001 | Task 002（目标） | 差距说明 |
|---------|----------|----------------|---------|
| 单文件信息提取 | ✅ 覆盖 | ✅ 保留 | Task 001 从单一 JSON 提取；002 从多个文件提取 |
| 多文件查找 | ❌ 不涉及 | ✅ 新增 | 001 只有 1 个输入文件；002 需 4-6 个文件 |
| 跨文件交叉核对 | ❌ 不涉及 | ✅ 新增 | 核心差异：找出一致、冲突、缺失的信息 |
| 证据引用 | ⚠️ 基础 | ✅ 加强 | 001 引用 ticket ID；002 需引用文件名 + 记录 ID |
| 信息完整性判断 | ❌ 不涉及 | ✅ 新增 | 识别哪些事实已确认、哪些需推断、哪些未知 |
| 干扰信息识别 | ❌ 不涉及 | ✅ 新增 | 1 个有效干扰文件，Agent 需判断是否相关 |
| 工具执行 | ❌ 不涉及 | ❌ 暂不引入 | 002 仅用 read dir / read file / write file |
| 编码能力 | ❌ 不涉及 | ❌ 暂不引入 | 仍属纯阅读理解 + 推理任务 |
| 错误修复 | ❌ 不涉及 | ❌ 暂不引入 | 不涉及代码调试 |
| 结构化交付 | ✅ 覆盖 | ✅ 保留 | 002 沿用 JSON + Markdown 格式，字段略增 |
| 长流程状态保持 | ❌ 不涉及 | ❌ 暂不引入 | 单次会话可完成 |
| 外部 API 调用 | ❌ 不涉及 | ❌ 不涉及 | 完全离线运行 |
| 跨文件时间线构建 | ❌ 不涉及 | ✅ 新增 | 002 要求按时间顺序组织事件 |

## 难度等级说明

```
Task 001                              Task 002
单一输入 → 单文件分析                   多文件输入 → 跨文件核对 → 综合判断
信息全、无冲突                         存在重复、冲突、缺失和干扰
证据引用简单                           证据引用需跨文件交叉验证
统计为主                               推理为主
```

## 执行方式变化

```
Task 001: 读 1 个文件 → 统计 → 输出
Task 002: 浏览目录 → 识别相关文件 → 跨文件读取 → 交叉核对 → 综合判断 → 输出
```

002 需要 Agent 主动扫描目录、筛选相关文件、比对多个来源，而非被动接收单一输入。

## 评分复杂度变化

Task 001 的评分重点是计数正确性和风险排序合理性，差异空间有限。Task 002 因为涉及跨文件推理，人工评分需要判断：
- Agent 是否找到了正确的文件子集（排除干扰文件）
- 跨文件核对是否完整（识别了一致、冲突、缺口）
- 根因推断是否有证据支撑（不能凭空猜测）
- 未知信息是否诚实标注（区分事实与推断）

这些新维度使评分表从 4 项 80 分扩展到 6 项 100 分，但仍是人工可稳定判断的范畴。
